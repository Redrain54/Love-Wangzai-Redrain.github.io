HTTP状态码总览(An Overview of HTTP Status Codes)

  • 1××：Informational Messages (give some information while the connection is still in progress)

    以1开头的状态码只是一个临时的，在HTTP/1.1时引入。Server可以发送一个Expect: 100-continue信息，告诉client继续发送剩下的请求，如果已经发送完毕就忽略。HTTP/1.0客户端会忽略这个header。

  • 2××：Successful (mean a successful connection was made)

    以2开头的状态码告诉client请求已被成功处理。如果是一个GET请求，server会在响应体中包含被请求的资源。

    • 200 OK：everything’s okay
    • 202 Accepted: 请求已被接受，但响应中可能不包含资源。这对于服务器端的异步处理非常有用。服务器可以选择发送信息进行监控。
    • 204 No Content: 响应中没有消息体。
    • 205 Reset Content: 让客户端重置其文档视图。
    • 206 Partial Content: 指响应只包含部分内容。

  • 3××：Redirection（重定位）

    以3开头的状态码表明需要客户端执行额外的操作。最常见的用例是为了获取资源而跳转到一个不同的URL。

    • 301 Move Permanently： 该资源现在位于一个新的URL。

    • 303 See Other: 该资源临时位于一个新的URL。Location响应头包含临时URL。

    • 304 Not Modified: 服务器已经确定资源没有改变，客户端应该使用它的缓存副本。这取决于客户端正在发送ETag(Enttity Tag 实体标签)信息，该信息是内容的哈希值。服务器将其与自己计算的ETag进行比较，以检查是否有修改。

      应用场景：浏览器第一次给服务器发送请求后，服务器会在响应头中加上Last-Modified字段（或Etag字段），浏览器接收到后若再次请求，会在请求头中携带If-Modified-Since（或If-None-Match）字段，这个字段的值是服务器传来的最后修改时间。服务器拿到这个字段之后会与该资源最后修改时间比较：

      • 如果请求头中的值小于最后修改时间，说明是时候更新了。就返回新的资源，跟常规的HTTP请求一样。
      • 否则返回304，告诉浏览器直接用缓存。

      Last-Modified与ETag对比

      1. 在精准度上，ETag优于Last-Modified。优于 ETag 是按照内容给资源上标识，因此能准确感知资源的变化。而 Last-Modified 就不一样了，它在一些特殊的情况并不能准确感知资源变化，主要有两种情况:
        • 编辑了资源文件，但是文件内容并没有更改，这样也会造成缓存失效。
        • Last-Modified 能够感知的单位时间是秒，如果文件在 1 秒内改变了多次，那么这时候的 Last-Modified 并没有体现出修改了。
      2. 在性能上，Last-Modified优于ETag，也很简单理解，Last-Modified仅仅只是记录一个时间点，而 Etag需要根据文件的具体内容生成哈希值。
      3. 如果两种方式都支持的话，服务器会优先考虑ETag。

  • 4xx: Client Error (show there was a client-side error)

    当服务器认为客户机出错时，就会使用这些代码，无论是请求无效的资源还是发出错误的请求。这个类中最流行的代码是404 Not Found,404表示该资源无效，在服务器上不存在。

    • 400 Bad Request: 请求格式不正确。
    • 401 Unauthorized: 请求都需要身份验证。客户端可以使用Authorization头重复请求。如果客户端已经包含了Authorization头，则凭据(credentials)是错误的。
    • 403 Forbidden: 服务器拒绝访问该资源。
    • 405 Method Not Allowed: 请求行中使用的无效HTTP verb，或者服务器不支持该verb。
    • 409 Conflict: 服务器无法完成请求，因为客户端试图修改比客户端时间戳更新的资源。冲突主要发生在对资源进行协作编辑期间的PUT请求。

  • 5xx: Server Error (indocate a server-side error, like a failed database connection, or the server being down)

    这类代码用于在处理请求时指示服务器故障。最常用的错误码是500 Internal Server Error。

    • 501 Not Implemented: 服务器还不支持所请求的功能
    • 503 Service Unavailable: 如果服务器上的内部系统失败或服务器超载，就会发生这种情况。通常，服务器甚至不会响应，请求会超时。

请求方法 (Verbs/Methods)

应该在主机上执行的操作是通过HTTP verbs指定的。当然，客户机希望主机执行一些操作。HTTP在一些基础上进行了规范化，这些基础捕获了普遍适用于各种应用程序的要点。(The action that should be performed on the host is specified via HTTP verbs. Of course, there are several actions that a client would like the host to perform. HTTP has formalized on a few that capture the essentials that are universally applicable for all kinds of applications.)

  • GET: fetch an existing resource. URL包含服务器定位和返回资源所需的所有必要信息。

  • POST: create a new resource. POST请求通常携带一个指定新资源数据的有效负载(payload)。（也就是有传参）

  • *PUT: update an existing resource. * 有效负载可能包含资源的更新数据。

  • DELETE: delete an existing resource.

    PUT and DELETE are sometimes considered specialized versions of the POST verb, and they may be packaged as POST requests with the payload containing the exact action: create, update or delete.

    下面三个不常用，但HTTP同样支持

  • HEAD: 类似于GET，但是没有消息主体。它用于检索特定资源的服务器头，通常通过时间戳检查资源是否发生了更改。

  • TRACE: 用于检索请求从服务器往返所需的跳数(the hops)。每个中间代理或网关将其IP或DNS名称注入Via报头字段。这可以用于诊断目的。

  • OPTIONS: 用于检索服务器功能。在客户端，可以使用它根据服务器支持的内容来修改请求。
